# Мини-игры: предметы, ингредиенты, награды и админка

Доработанная логика трёх мини-игр с единой системой предметов/ингредиентов, способами их получения (чтение, ежедневные задания, лидерборд) и администрированием.

---

## 1. Система предметов и инвентаря

### 1.1 Типы предметов

| Тип | Описание | Примеры |
|-----|----------|---------|
| `material` | Ингредиенты для алхимии, расходники для культивации | Духовная трава, Камень духа, Корень мандрагоры |
| `consumable` | Одноразовое использование (будущее: баффы) | Пилюля маны, Золотая пилюля |
| `special` | Редкие, для отображения в профиле/коллекции | Печать стихии, Трофей лидерборда |

### 1.2 Схема GameItem (коллекция)

```ts
// schemas/game-item.schema.ts
{
  id: string;           // уникальный ключ (slug), например "spirit_stone"
  name: string;         // "Камень духа"
  description?: string;
  icon: string;         // URL или имя иконки
  type: 'material' | 'consumable' | 'special';
  rarity: 'common' | 'uncommon' | 'rare' | 'epic' | 'legendary';
  stackable: boolean;   // default true для material
  maxStack: number;      // default 999
  usedInRecipes: boolean; // для алхимии: участвует в рецептах
  sortOrder: number;    // для отображения в админке/инвентаре
  isActive: boolean;    // выключенные не дропаются и не показываются
}
```

### 1.3 Инвентарь пользователя (user.schema)

```ts
// В User добавить:
inventory: {
  itemId: string;   // id из GameItem
  count: number;
}[];

// Опционально для игр и профиля:
element?: 'fire' | 'water' | 'earth' | 'wood' | 'metal';  // стихия в профиле
guildId?: Types.ObjectId;   // для будущих гильдий
lastPillCraftedAt?: Date;
lastWheelSpinAt?: Date;

// Игра «Учитель — ученики»: список учеников (персонажи из тайтлов) и их боевые статы
disciples?: {
  characterId: Types.ObjectId;   // ref Character
  titleId: Types.ObjectId;      // для отображения тайтла
  recruitedAt: Date;
  attack: number;                // боевые характеристики (генерируются при найме)
  defense: number;
  speed: number;
  hp: number;                    // или «дух» — условные очки для боя
}[];
maxDisciples?: number;           // слоты (например 5), настраивается
lastTrainingAt?: Date;         // кулдаун одной тренировки в день (тренируется один выбранный ученик)
combatRating?: number;          // рейтинг для матчмейкинга (пересчитывается от статов учеников)
lastBattleAt?: Date;            // опционально: лимит боёв в день
lastRerollCandidate?: {         // последний кандидат с реролла (для recruit)
  characterId: Types.ObjectId;
  titleId: Types.ObjectId;
  attack: number;
  defense: number;
  speed: number;
  hp: number;
  at: Date;
}

// Счётчики для дропа за чтение (сбрасывать по началу дня UTC):
readingDropsDate?: Date;           // начало дня, за который считаем
readingChaptersToday?: number;     // глав прочитано за этот день
readingDropsToday?: { itemId: string; count: number }[];  // лимит maxDropsPerDay по каждому предмету
```

Хелпер: `addToInventory(user, itemId, count)` с учётом `maxStack` и объединением стаков.

## 2. Получение предметов (дропы)

### 2.1 За чтение глав

- **Момент:** при успешном добавлении главы в историю чтения (то же место, где начисляется XP и вызывается `incrementDailyQuestProgress`), после проверки на бота.
- **Логика:** раз в день на первую за день прочитанную главу не дропаем; на каждую следующую главу — один ролл по таблице дропа (см. ниже). Ограничение: не более **N дропов в день** (N настраивается в админке глобально или per-rule).
- **Таблица дропов (админка):** сущность `ReadingDropRule`:
  - `itemId`, `chance` (0–1), `minChaptersToday` (после скольких глав за день возможно), `maxDropsPerDay` (глобальный лимит по этому предмету в день на пользователя), `isActive`.

Пример: «Камень духа» — chance 0.15, minChaptersToday 1, maxDropsPerDay 3. То есть с 2-й главы за день при каждом прочтении главы ролл 15%; не более 3 камней в день.

### 2.2 За выполнение ежедневных заданий

- **Момент:** при `claimDailyQuest` (после начисления XP и монет).
- **Логика:** к награде квеста добавляется выдача предметов по таблице «награды за тип квеста».
- **Таблица (админка):** сущность `DailyQuestItemReward`:
  - `questType` (read_chapters, add_bookmark, leave_comment, rate_title, daily_login), `itemId`, `countMin`, `countMax` (если равны — фиксированное количество), `chance` (0–1, опционально: если 1 — всегда даём). Сортировка по приоритету (например, сначала фиксированные, потом случайные).

Пример: за «Прочитайте 5 глав» при получении награды дополнительно даём 1–2 «Духовная трава» (countMin 1, countMax 2, chance 1).

### 2.3 Бонусы из лидерборда

- **Момент:** не при просмотре лидерборда, а по расписанию (cron): раз в неделю и/или раз в месяц пересчитываем позиции и начисляем награды за прошлый период.
- **Логика:**
  - Храним срез «топ за неделю/месяц» на конец периода (или пересчитываем по данным за период: chaptersRead, ratings и т.д. — как уже считается в `getLeaderboardByPeriod`).
  - По таблице наград лидерборда для каждой позиции (или диапазона) выдаём предметы и/или монеты в инвентарь/баланс.
- **Таблица (админка):** сущность `LeaderboardReward`:
  - `category` (level, readingTime, ratings, comments, streak, chaptersRead), `period` (week, month), `rankMin`, `rankMax` (включительно), `itemId` (опционально), `itemCount`, `coins` (опционально), `isActive`.

Пример: категория `chaptersRead`, период `week`, места 1–3: 5 «Камень духа», 50 монет; места 4–10: 2 «Камень духа», 20 монет.

- **Крон:** например, по понедельникам в 00:05 — награды за неделю; 1-го числа в 00:05 — за месяц. Один и тот же пользователь может получить награду и за неделю, и за месяц, если попал в оба среза.

## 3. Игра 1: Учитель и ученики (тренировки + PvP)

Концепция: пользователь — «наставник», который берёт себе учеников из мира тайтлов (персонажи из коллекции Character), тренирует их и выставляет на бои с учениками других игроков. У каждого ученика свои характеристики; матчмейкинг по боевому потенциалу и рандом в бою уравнивают шансы.

---

### 3.1 Набор учеников (реролл за монеты)

- **Источник персонажей:** случайные персонажи из тайтлов платформы (коллекция Character, статус approved). Можно ограничить пул: только из тайтлов, которые есть в закладках пользователя, или глобально из всех тайтлов (настраивается в админке).
- **Реролл:** пользователь платит **монеты** за один «призыв» и получает **одного случайного персонажа** как кандидата в ученики. Можно принять его в команду или отказаться и рероллить снова (снова платить).
- **Лимит слотов:** у пользователя не более **N учеников** (например 5). Поле `maxDisciples` в конфиге или на пользователе. Освободить слот можно «отпустив» ученика (без возврата монет).
- **Уникальность:** одного и того же персонажа (characterId) нельзя взять дважды в одну команду; у разных пользователей один и тот же персонаж из тайтла может быть у нескольких наставников.
- **Характеристики при найме:** при принятии персонажа в ученики сервер генерирует ему **боевые статы** (attack, defense, speed, hp) по правилам (см. ниже), чтобы разные ученики отличались по потенциалу.

### 3.2 Боевые характеристики ученика

- У каждого ученика в `user.disciples[]` хранятся: `characterId`, `titleId`, `recruitedAt`, `attack`, `defense`, `speed`, `hp`.
- **Генерация при найме (админ-настраиваемые диапазоны):**
  - Каждая характеристика — случайное число в диапазоне, например: attack [5–15], defense [5–15], speed [3–12], hp [20–50]. Итоговый **боевой потенциал** (Combat Power, CP) считаем по формуле, например:  
    `CP = attack*1.2 + defense*1.0 + speed*0.8 + hp*0.3` (коэффициенты в конфиге).
  - Можно добавить редкость персонажа (по role в Character: main/supporting дают небольшой бонус к диапазону) — опционально.

### 3.3 Тренировка учеников

- **Ресурс:** монеты за одну тренировку (например 15 монет). Опционально расход предмета «Камень духа» для снижения стоимости или бонуса к приросту.
- **Лимит:** одна тренировка **раз в 24 часа** (по `lastTrainingAt`). За одну тренировку прокачивается **один выбранный ученик**.
- **Выбор ученика:** клиент передаёт `disciplineIndex` или `characterId` ученика, которого тренируем.
- **Рандом прироста (конфиг в админке):** как раньше — исходы small/medium/large с весами; прирост к статам выбранного ученика (например small: +1 к одной случайной характеристике, medium: +1–2 к двум, large: +2–3 к одной или +1 ко всем). Не превышать максимумы по статам (админ задаёт cap, например 50 по каждой).
- **Стихия в профиле:** при выбранной `element` у пользователя — небольшой модификатор к весам (шанс large выше). После тренировки пересчитываем `combatRating` пользователя по новой сумме CP учеников.

### 3.4 Бои с другими игроками (PvP)

- **Матчмейкинг:** рейтинг игрока — `combatRating` (сумма CP всех учеников или, например, сумма CP топ-3 учеников, если в бою участвуют не все — настраивается). При поиске противника выбираем пользователя с **близким combatRating** (например ±15% или по «бакетам» рейтинга). Очередь: либо подбор в реальном времени, либо «найти противника» — сервер возвращает одного подходящего соперника из недавно искавших или из таблицы по рейтингу.
- **Формат боя:** например **1v1** (один ученик с каждой стороны) или **3v3** (три ученика). Кто участвует: пользователь выбирает из своих учеников до начала боя (или автоматически топ-3 по CP). У противника — аналогично его выбранные/топ ученики.
- **Исход боя:** чтобы бои не были «у кого больше CP — тот и победил», вводится **рандом**:
  - Вероятность победы стороны A против B: например `winChance = 0.5 + k * (CP_A - CP_B) / (CP_A + CP_B)`, где `k` — коэффициент (например 0.3), так что при равном CP шансы 50/50, при перевесе 20% по CP — около 56/44. Конкретную формулу задать в конфиге.
  - Либо пошаговый упрощённый бой (каждый ход наносится урон по формуле attack vs defense, рандом по speed кто бьёт первым) — тогда рандом заложен в криты и порядок ходов.
- **Награды:** за победу — монеты и/или XP (и опционально предметы); за поражение — меньше или только участие. Не менять рейтинг или менять мягко (±5–10 очков), чтобы не создавать гриндинг рейтинга.
- **Лимит боёв:** опционально не более N боёв в день (по `lastBattleAt` и счётчику боёв за день).

### 3.5 Баланс и равенство боевого потенциала

- Реролл даёт разных по статам персонажей — кто-то сильнее, кто-то слабее; тренировка выравнивает и усиливает своих. Матчмейкинг по `combatRating` сводит в пару тех, у кого примерно равная суммарная сила.
- Рандом в расчёте победы (или пошаговый бой с критами) не даёт гарантированной победы «киту», так что бои выглядят честнее.
- Админка: настройка диапазонов статов при найме, формулы CP и winChance, лимиты слотов, стоимости реролла и тренировки, лимиты боёв в день.

### 3.6 API (игра «Учитель — ученики»)

- `GET /users/profile/disciples` — список своих учеников (с подгрузкой Character: name, avatar, titleId → название тайтла), их статы, combatRating, можно ли тренироваться/искать бой.
- `POST /users/profile/disciples/reroll` — заплатить монеты, получить одного случайного персонажа (characterId, name, avatar, titleId) **как кандидата** с предпросчитанными статами (не сохранять в команду). Ответ: кандидат + сгенерированные attack/defense/speed/hp. Клиент показывает «взять в ученики / реролл ещё».
- `POST /users/profile/disciples/recruit` — body `{ characterId: string }`. Принять последнего показанного кандидата в команду. На сервере хранить **последнего кандидата реролла** (в сессии/памяти или в пользователе: `lastRerollCandidate: { characterId, titleId, attack, defense, speed, hp, at: Date }`). При recruit проверять, что `characterId` совпадает с `lastRerollCandidate.characterId` и запрос не старше 5–10 минут — тогда сохранять ученика с теми же статами и очищать кандидата. Монеты списываются при реролле.
- `POST /users/profile/disciples/dismiss` — body `{ characterId: string }` или индекс. Убрать ученика из команды (освободить слот).
- `POST /users/profile/disciples/train` — body `{ characterId: string }` или индекс. Проверка кулдауна и баланса, ролл прироста, обновление статов одного ученика, пересчёт combatRating, `lastTrainingAt`.
- `GET /users/profile/disciples/battle-match` — подбор противника (по combatRating). Ответ: противник (userId, username, avatar, список учеников в формате для боя — статы, имена, без characterId если не нужно), его combatRating.
- `POST /users/profile/disciples/battle` — body `{ opponentUserId: string, myDiscipleIds: string[], opponentDiscipleIds?: string[] }`. Провести бой: рассчитать исход по формуле/пошагово, вернуть результат (win/lose, награды), обновить счётчики и при необходимости рейтинг.

## 4. Игра 2: Алхимия пилюль

### 4.1 Рецепты (админка)

- Сущность `AlchemyRecipe`:
  - `id`, `name`, `description`, `icon`, `coinCost` (опционально),
  - `ingredients`: `{ itemId, count }[]`,
  - `resultType`: `'pill_common' | 'pill_quality' | 'pill_legendary'` (или привязка к конкретному `consumable` itemId),
  - `qualityWeights`: веса для рандома качества при успешной варке, например `{ common: 70, quality: 25, legendary: 5 }` (в процентах или долях).

### 4.2 Получение ингредиентов

- Из дропов за чтение и за ежедневные задания (и при желании из лидерборда). В админке настраиваются конкретные `GameItem` с `usedInRecipes: true` и правила дропа (см. выше).

### 4.3 Лимиты

- Одна варка в день (по `lastPillCraftedAt`); можно ограничить «одна варка в день по любому рецепту» или «N варок в день» — в конфиге.

### 4.4 Логика варки

1. Проверка: кулдаун, наличие ингредиентов и монет (если `coinCost`).
2. Списание ингредиентов и монет.
3. Ролл по `qualityWeights` → качество пилюли.
4. Эффект по качеству (настраивается):
   - common: например +5 XP или +3 монеты;
   - quality: +15 XP или +10 монет или выдача предмета из пула;
   - legendary: редкая декорация или крупный бонус монет/XP.
5. Обновление `lastPillCraftedAt`, начисление наград пользователю.

### 4.5 Гильдии (будущее)

- Поле `guildId` у пользователя. В настройках гильдии (или глобально) — бонус к шансу качества при варке (например +1% к quality/legendary за уровень гильдии). Пока можно не реализовывать, только заложить поле.

### 4.6 API

- `GET /users/profile/alchemy/recipes` — список доступных рецептов (активных) с текущим наличием ингредиентов у пользователя.
- `GET /users/profile/alchemy/status` — можно ли варить сегодня, lastPillCraftedAt.
- `POST /users/profile/alchemy/craft` — body `{ recipeId: string }`. Валидация, списание, ролл, награда.

## 5. Игра 3: Колесо судьбы (рулетка)

### 5.1 Сегменты (админка)

- Сущность `WheelSegment` (или один документ с массивом сегментов):
  - `rewardType`: 'xp' | 'coins' | 'item' | 'element_bonus' | 'empty',
  - `weight` (для рандома),
  - `param`: для xp — число; для coins — число; для item — itemId и count; для element_bonus — множитель к следующему сегменту или отдельная награда «бонус стихии» (доп. монеты/предмет).
  - `label` (для отображения на клиенте, например «10 монет»).

### 5.2 Стоимость и лимит

- Один спин в день, стоимость в монетах (например 20) — в конфиге.

### 5.3 Стихия

- Если у пользователя задана `element`, при выпадении сегмента `element_bonus` выдаётся улучшенная награда (например, +5 монет к выпавшим или фиксированный предмет «печать стихии»). Либо отдельный сегмент «бонус стихии» с небольшим весом.

### 5.4 API

- `GET /users/profile/wheel` — сегменты (без весов), стоимость, можно ли крутить, lastWheelSpinAt.
- `POST /users/profile/wheel/spin` — списание монет, кулдаун, ролл по весам, выдача награды (XP/монеты/предмет в инвентарь), обновление lastWheelSpinAt.

## 6. Администрирование

### 6.1 Предметы (GameItem)

- **Модуль:** отдельный модуль `game-items` (или `minigames`) с админ-контроллером.
- **Эндпоинты (admin):**
  - `GET /game-items/admin` — список с фильтром по type, rarity, isActive.
  - `GET /game-items/admin/:id` — один предмет.
  - `POST /game-items/admin` — создание (body: id, name, type, rarity, …).
  - `PATCH /game-items/admin/:id` — обновление.
  - `DELETE /game-items/admin/:id` — мягкое отключение (isActive: false) или удаление.

### 6.2 Дропы за чтение

- **Эндпоинты (admin):**
  - `GET /game-items/admin/drops/reading` — список правил ReadingDropRule.
  - `POST /game-items/admin/drops/reading` — добавить правило (itemId, chance, minChaptersToday, maxDropsPerDay).
  - `PATCH /game-items/admin/drops/reading/:id` — изменить.
  - `DELETE /game-items/admin/drops/reading/:id` — удалить.

### 6.3 Награды за ежедневные задания

- **Эндпоинты (admin):**
  - `GET /game-items/admin/drops/daily-quest` — список DailyQuestItemReward (по questType).
  - `POST /game-items/admin/drops/daily-quest` — привязать награду к типу квеста (questType, itemId, countMin, countMax, chance).
  - `PATCH` / `DELETE` по id.

### 6.4 Награды лидерборда

- **Эндпоинты (admin):**
  - `GET /game-items/admin/rewards/leaderboard` — список LeaderboardReward (фильтр по category, period).
  - `POST /game-items/admin/rewards/leaderboard` — (category, period, rankMin, rankMax, itemId?, itemCount, coins?).
  - `PATCH` / `DELETE` по id.

### 6.5 Игра «Учитель — ученики» (дисциплины, бои)

- **Эндпоинты (admin):**
  - `GET /game-items/admin/config/disciples` — конфиг: стоимость реролла (монеты), стоимость тренировки, maxDisciples, лимит боёв в день; диапазоны статов при найме (attack, defense, speed, hp — min/max); формула CP (коэффициенты); параметры winChance (k); лимиты статов (cap) после тренировок; время жизни кандидата реролла (минуты); пул персонажей (все approved / только из закладок пользователя).
  - `PUT /game-items/admin/config/disciples` — обновить конфиг.

### 6.6 Алхимия

- **Эндпоинты (admin):**
  - `GET /game-items/admin/recipes` — список рецептов.
  - `POST /game-items/admin/recipes` — создать рецепт (ingredients, resultType, qualityWeights, coinCost).
  - `PATCH` / `DELETE` по id.

### 6.7 Колесо

- **Эндпоинты (admin):**
  - `GET /game-items/admin/config/wheel` — сегменты и стоимость спина.
  - `PUT /game-items/admin/config/wheel` — обновить массив сегментов и стоимость.

### 6.8 Выдача предметов пользователю (админ)

- `POST /game-items/admin/grant` — body `{ userId: string, itemId: string, count: number }` — для тестов и акций. Логировать в AdminLog.

## 7. Интеграция в существующий код

### 7.1 Чтение глав (users.service)

- В методе, где обновляется reading history и вызывается `incrementDailyQuestProgress(userId, 'read_chapters', 1)`, после этого вызвать сервис дропов: `this.dropService.tryReadingDrops(userId, isNewChapter)`.
- **Счётчики за день:** либо считать «глав прочитано сегодня» по readingHistory (по дате readAt за сегодня) в момент вызова, либо хранить на пользователе:
  - `readingDropsDate: Date` (начало дня UTC),
  - `readingChaptersToday: number`,
  - `readingDropsToday: { itemId: number }` (сколько раз выдали каждый предмет за день).
  При смене дня (getStartOfDayUTC() > readingDropsDate) обнулять счётчики. Так проще соблюдать maxDropsPerDay и minChaptersToday без тяжёлых агрегаций по истории.

### 7.2 Ежедневные задания (claimDailyQuest)

- После начисления XP и монет вызвать `this.dropService.grantDailyQuestRewards(userId, questType)` и выдать предметы по таблице DailyQuestItemReward для этого questType. Добавить в ответ поля `itemsGained?: { itemId, count }[]`.

### 7.3 Лидерборд (cron)

- Новый сервис или метод в users: раз в неделю/месяц получить срез топа за прошлый период (можно переиспользовать логику агрегации из getLeaderboardByPeriod за прошлую неделю/месяц), для каждого userId в топе прочитать LeaderboardReward и вызвать `addToInventory` / `addBalance`.

### 7.4 Профиль пользователя

- В ответы профиля (findProfileById, getProfileWithPrivacy) добавить: `inventory` (список с itemId, count, name, icon от GameItem), `element`, данные игры «Учитель — ученики» (disciples с подгрузкой имён/аватаров персонажей, combatRating), флаги «можно ли тренировать/искать бой/варить/крутить сегодня».

## 8. Сводная таблица сущностей

| Сущность | Назначение |
|----------|------------|
| GameItem | Справочник предметов (материалы, расходники, особые). |
| User.inventory | Инвентарь: itemId + count. |
| User.element, disciples, lastRerollCandidate, lastTrainingAt, combatRating, lastBattleAt, lastPillCraftedAt, lastWheelSpinAt | Состояние игр (ученики, тренировки, бои, алхимия, колесо). |
| ReadingDropRule | Дроп предмета за чтение (chance, лимиты). |
| DailyQuestItemReward | Награда предметами за тип квеста при claim. |
| LeaderboardReward | Награды за место в лидерборде (cron). |
| DisciplesConfig | Стоимость реролла/тренировки, maxDisciples, диапазоны статов, формула CP, winChance, лимиты. |
| AlchemyRecipe | Рецепт: ингредиенты, результат, веса качества. |
| WheelConfig | Сегменты (веса, типы наград), стоимость спина. |

Все перечисленные сущности администрируются через отдельный модуль с admin-контроллерами и защитой по роли `admin`.

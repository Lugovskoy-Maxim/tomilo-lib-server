# План: синхронизация с клиентом и устранение багов

На основе обзора кодовой базы **tomilo-lib-server** (NestJS, MongoDB, WebSocket, парсер манги, магазин, достижения, подписки и т.д.).

## 2. Синхронизация с клиентом

### 2.1 Единый контракт API

- **Документ:** `docs/api-standards.md` задаёт формат `ApiResponseDto<T>`: `success`, `data`, `message`, `errors`, `timestamp`, `path`, `method`.
- **Проблема:** Часть эндпоинтов при ошибках не бросает исключения, а возвращает объект с `success: false` вручную (например, `comments.controller.ts`, `reports.controller.ts`). Глобальный `HttpExceptionFilter` добавляет `path` и `method` только к выброшенным исключениям.
- **Действие:** Привести обработку ошибок к одному варианту: либо везде бросать `HttpException` и полагаться на фильтр, либо гарантировать, что ручные ответы с ошибками тоже содержат `path` и `method` в том же формате, что и в стандарте. Предпочтительно — унифицировать на «throw + filter», чтобы клиент всегда получал один и тот же формат ошибок.

### 2.2 WebSocket

- **NotificationsGateway** (`/api/notifications`): JWT из cookie/query/header, комната `user:${userId}`. События: `unread_count`, `progress`, `notification`. Клиенту нужно подключаться с тем же токеном, что и для REST.
- **ParsingGateway** (`/api/parsing`): без авторизации; события прогресса импорта. Риск: прогресс реализован через перезапись методов `MangaParserService` (monkey-patch). При двух одновременных сессиях вторая перезапишет обработчики — прогресс первой сессии может «сломаться».
- **Действие:** Документировать для клиента: имена событий, формат payload, способ передачи JWT для notifications. Для ParsingGateway — вынести прогресс в отдельный слой (например, callback/observer или отдельный сервис с привязкой по `sessionId`), чтобы не мутировать общий сервис.

### 2.3 Кэш и актуальность данных

- Кэш используется в `AppService`, `UsersService`, `TitlesService`, `ChaptersService` и др. При изменении данных (например, тайтл, глава, профиль) инвалидация не везде может быть явно видна.
- **Действие:** Проверить ключи кэша и инвалидацию при обновлении/удалении сущностей; при необходимости добавить явную инвалидацию после записи. Это нужно для того, чтобы клиент после действий (лайк, закладка, прогресс) получал актуальные данные при следующем запросе.

### 2.4 CORS и префикс API

- Глобальный префикс: `/api`. CORS настроен на конкретные origin (localhost, tomilo-lib.ru, IP). Новый фронт или домен потребует правки кода.
- **Действие:** Вынести разрешённые origin в конфиг (env/конфиг-модуль), чтобы добавлять клиентские домены без деплоя.

## 3. Устранение багов и улучшения

### 3.1 Критичные / заметные

| № | Где | Проблема | Решение |
|---|-----|----------|---------|
| 1 | `users.controller.ts` ~1305 | `isFriend = false` захардкожен (TODO: проверка дружбы) | Ввести модуль «друзья» или явно документировать, что режим «только для друзей» пока приравнивается к «никому». Пока оставить `false`, но не считать друзьём — т.е. профили с «friends» по сути скрыты для всех, кроме себя. |
| 2 | `parsing.gateway.ts` | Monkey-patch методов `MangaParserService` для прогресса; при двух клиентах прогресс может путаться | Вынести отправку прогресса в отдельный механизм (сервис прогресса по `sessionId` или коллбэки в методах парсера), не перезаписывать методы сервиса. |
| 3 | Comments/Reports | Ошибки возвращаются вручную (`success: false`) без единого формата с `HttpExceptionFilter` | Перейти на выброс `HttpException` в catch и использовать общий фильтр; убедиться, что в ответе есть `path`, `method`, `errors` как в api-standards. |

### 3.2 Конфигурация и безопасность

| № | Где | Проблема | Решение |
|---|-----|----------|---------|
| 4 | JWT | `jwt.config.ts` есть и используется в `titles.controller`, `jwt.util`, стратегии; в стратегии для лога всё ещё читается `process.env.JWT_SECRET` напрямую | Оставить один источник правды — `getJwtSecret()` из `jwt.config.ts`; в стратегии для лога использовать результат `getJwtSecret()` (например, проверять «есть ли значение» без вывода секрета). Добавить `jwt.config.ts` в репозиторий (если ещё не закоммичен). |
| 5 | CORS | Список origin захардкожен в `main.ts` | Вынести в конфиг (env или ConfigService), документировать переменную для новых клиентов. |

### 3.3 Надёжность и наблюдаемость

| № | Где | Проблема | Решение |
|---|-----|----------|---------|
| 6 | Throttler | Глобальный лимит (1000 req/min); нет различий по маршрутам/ролям | По необходимости ввести отдельные лимиты для тяжёлых эндпоинтов (парсинг, загрузка файлов, авторизация) и/или для гостей vs авторизованных. |
| 7 | Ошибки | В ответах ошибок от фильтра нет поля `data`; в api-standards оно опционально | Оставить как есть или явно зафиксировать в стандарте: при ошибке `data` не возвращается. |

## 4. Порядок работ (приоритеты)

1. **Синхронизация с клиентом**
   - Унифицировать формат ошибок (п. 2.1, 3.3).
   - Документировать WebSocket (события, авторизация) и при необходимости вынести CORS в конфиг (п. 2.2, 2.4).

2. **Баги**
   - Убрать monkey-patch в ParsingGateway (п. 3.1, №2).
   - В comments/reports перейти на throw + HttpExceptionFilter (п. 3.1, №3).
   - Зафиксировать использование JWT только через `getJwtSecret()` и добавить `jwt.config.ts` в репозиторий (п. 3.2, №4).

3. **Дальше**
   - Модуль друзей или явное описание поведения «только для друзей» (п. 3.1, №1).
   - Проверка инвалидации кэша при изменениях данных (п. 2.3).
   - Настройка CORS из конфига (п. 3.2, №5) и при необходимости точечный throttling (п. 3.3, №6).

После выполнения п. 1–2 клиент сможет опираться на единый контракт ответов и стабильный прогресс по WebSocket; сервер станет проще сопровождать и расширять.
